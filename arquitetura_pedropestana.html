secção 02

Visão Geral e Filosofia

O pedropestana.com é um site estático puro — HTML, CSS e JavaScript vanilla, sem framework de frontend, sem sistema de build, sem dependências de runtime. O objectivo foi minimizar a complexidade operacional e maximizar a longevidade: um site que funciona sem servidor, sem base de dados, sem processo de compilação e sem custos de hosting.

O conteúdo dinâmico (artigos, projectos, competências) é gerido através de ficheiros JSON em /data/ carregados via fetch() no browser. O CMS é o Sveltia — uma interface web que edita esses ficheiros JSON directamente no repositório GitHub via OAuth. Não há backend.

Princípio de design
Zero custo de hosting · Zero dependências de runtime · Zero build step · Editável via browser (Sveltia CMS) · Deployável com git push · SEO-ready · GDPR-compliant

Audiência alvo

  • Clientes de consultoria B2B (geoespacial, dados, Django)
  • Recrutadores internacionais (mercado AU, UK, PT)
  • Pares técnicos que lêem os artigos de investigação
  • Referência de portfólio para propostas TerraDigital

Camada de Dados — Ficheiros JSON

Todo o conteúdo dinâmico do site é armazenado em ficheiros JSON em /data/. O JavaScript carrega estes ficheiros via fetch() e renderiza o HTML no cliente. Não há servidor, não há API, não há base de dados.

Tipos de blocos de artigo suportados

  • heading1 / heading2 / heading3 — secções
  • paragraph — texto corrido
  • authors / keywords — metadados académicos
  • code — bloco de código monospace fundo escuro
  • formula — equação matemática em caixa destacada
  • caption — legenda de figura/listagem
  • list — lista não ordenada
  • table — tabela com headers e rows
  • references — lista numerada de referências bibliográficas

Ordenação de artigos (js/main.js)

Artigos ordenados por: (1) featured primeiro, (2) id descendente. Esta ordenação é independente da língua — resolve o problema de new Date("Maio 2026") retornar Invalid Date em JavaScript, que causava ordenação incorrecta na versão PT.

secção 06

Sveltia CMS

Interface web para edição dos ficheiros JSON directamente no repositório GitHub. Acessível em /admin/ após autenticação OAuth.

Autenticação

  • GitHub OAuth App — client registado em github.com/settings/developers
  • Cloudflare Workers proxy — sveltia-cms-auth.eu-cac.workers.dev
  • Repositório: ppestana/github.io · branch: main

Colecções configuradas

ColecçãoLabel CMSFicheiros geridos

articlesArticles / Artigosdata/articles_en.json · data/articles_pt.json
skillsSkills / Competênciasdata/skills_en.json · data/skills_pt.json
Limitação actual
O config.yml não inclui o campo content (array de blocos) na colecção articles. Artigos com conteúdo completo têm de ser editados directamente no JSON via editor de código.
secção 07

JavaScript — Arquitectura

js/main.js

  • Detecção de língua via document.documentElement.lang
  • i18n — objecto com strings EN/PT, função t(key)
  • Carregamento de artigos — fetch JSON → ordenar (featured first, id desc) → renderizar cards
  • Card de artigo em destaque — badge FEATURED, sem painel visual lateral
  • Carregamento de competências — fetch JSON → renderizar categorias
  • Navegação mobile — hamburger menu toggle

js/article-details.js

  • Lê parâmetro ?article=N do URL
  • Fetch do JSON correspondente à língua da página
  • Renderização de blocos via switch em block.type — suporta: heading1/2/3, paragraph, authors, keywords, formula, code, caption, list, table, references, image (★ novo 31 Mai), image-row (★ novo 31 Mai)
  • image — imagem única com legenda opcional, border-radius + box-shadow
  • image-row — duas imagens lado a lado (flex), responsivo em mobile
  • Layout: coluna centrada max-width 820px, sem sidebar
  • Artigos relacionados no rodapé em grid horizontal

js/cookie-consent.js

  • Banner GDPR — aparece na primeira visita
  • Preferências em localStorage
  • GA4 (G-THSWRD90CK) só carrega após consentimento explícito
  • Link "repor preferências" via atributo data-cookie-settings

SEO — Search Engine Optimization

Implementado em 29 de Maio de 2026. Todas as páginas EN e PT têm o conjunto completo de meta tags.

Tags implementadas em cada página

  • <title> — optimizado com palavras-chave relevantes (PostGIS, Django, GIS, Portugal)
  • <meta name="description"> — descrição única por página, 150–160 caracteres
  • <meta name="author"> — Pedro Pestana
  • <meta name="robots"> — index, follow
  • <meta property="og:title"> — título Open Graph
  • <meta property="og:description"> — descrição Open Graph
  • <meta property="og:type"> — website
  • <meta property="og:url"> — URL canónico da página
  • <meta property="og:image"> — /assets/profile.jpg
  • <meta property="og:locale"> — en_GB ou pt_PT conforme a versão
  • <meta property="og:site_name"> — Pedro Pestana · TerraDigital
  • <link rel="alternate" hreflang> — EN/PT em todas as páginas
  • <link rel="canonical"> — URL canónico

sitemap.xml

  • 24 URLs — 8 EN + 8 PT + 8 artigos publicados (EN+PT · ids 1, 3, 4, 5)
  • Prioridades: homepage 1.0, about/work/articles 0.9, skills/contact/cv 0.7–0.8, privacy 0.3
  • Frequências: homepage/articles weekly, restantes monthly ou yearly
  • Artigo id:2 (in-preparation) excluído intencionalmente

robots.txt

  • User-agent: * · Allow: /
  • Referência ao sitemap: Sitemap: https://www.pedropestana.com/sitemap.xml
Passo seguinte recomendado
Submeter o sitemap.xml no Google Search Console em search.google.com/search-console para acelerar a indexação. Requer verificação de propriedade do domínio (via DNS TXT record no Namecheap ou via ficheiro HTML na raiz do repo).
